# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A French-language toolkit for learning modern Hebrew, deployed as static files on GitHub Pages at `https://rubischtgadol.github.io/flashcards-hebreu/`. There are **no dependencies, no tests, no package manager** ŌĆö every deployed file is a single self-contained HTML document with inline CSS and vanilla JS. The only tooling is `build.js` (dev-only, zero-dependency Node script, not deployed). To develop, open the files in a browser; to deploy, replace files on the `main` branch (GitHub Pages redeploys automatically, same URLs).

Important: `index.html` uses `fetch()`, so it must be served over HTTP, not opened as a `file://`. Run a local server from the repo root (e.g. `python3 -m http.server`) and open `http://localhost:8000/`.

## The files

- **`vocabulaire_hebreu.html`** ŌĆö the grammar + vocabulary notebook. This is the **single source of truth** for all content. Editing vocabulary means editing this file.
- **`index.html`** ŌĆö the online flashcards app (the main page). It does **not** contain vocabulary; at load it fetches `vocabulaire_hebreu.html` and extracts the cards at runtime. It is also the **source of the standalone file**: the network-only code (fetch + runtime extraction) is fenced by `BUILD:ONLINE-ONLY` marker comments that `build.js` relies on.
- **`flashcards_hebreu.html`** ŌĆö the standalone offline flashcards. **100% generated by `build.js` ŌĆö never edit it by hand.** It is `index.html` with the `BUILD:ONLINE-ONLY` block replaced by an embedded `const CARDS = [...]` snapshot and a direct boot (no loader).
- **`build.js`** ŌĆö regenerates `flashcards_hebreu.html` from the two files above, printing the per-section card count and failing if an expected section drops to 0. `node build.js --check` verifies without writing. **Run it after any edit to the notebook or to `index.html`.** It re-implements `extractCards()` with regex parsing (no DOM in Node) ŌĆö any change to `extractCards()` in `index.html` must be mirrored there, and `--check` byte-compares so drift is caught.
- **`manifest.webmanifest`, `sw.js`, `icons/`** ŌĆö the PWA layer (install + offline). The service worker registration lives **inside** the `BUILD:ONLINE-ONLY` block of `index.html` and must stay there (the standalone file must not register it). In `sw.js`, only root navigations are collapsed onto the `./` shell ŌĆö deep pages like the notebook must be served as-is. Bump `VERSION` in `sw.js` after changing its strategy, the asset list, or the icons. The `:root` design-token block must stay **identical** between the notebook and `index.html` (notebook is the reference); if `--bg`/`--gold` change, regenerate the icons and update `manifest.webmanifest`/`theme-color`.

## The extraction coupling (most important thing to understand)

`index.html`'s `extractCards(doc)` (near line 1056) scrapes the notebook's DOM. Its parsing is tightly coupled to the notebook's markup, so edits to `vocabulaire_hebreu.html` can silently drop cards if they break these assumptions:

- Sections are found by `<h2>` elements containing a `<span class="count">LABEL</span>`. The **exact `LABEL` text** is the key used throughout `extractCards` (e.g. `'Verbes'`, `'Adjectifs'`, `'Noms'`, `'Pronoms personnels'`, `'Nombres (0ŌĆō10)'`). Renaming a count label detaches its section from extraction.
- Word entries live in either `<table><tbody><tr>` rows (Verbes, Adjectifs, Noms) or `<ul class="word-list"><li>` items (pronouns, prepositions, numbers, expressions, **Phrases**, etc. ŌĆö see the `listCats` map). The `Phrases` section holds full example sentences (cat `Phrases`); they flow through every mode as ordinary cards and get a smaller `.big-he.phrase`/`.big-fr.phrase` size so long sentences wrap cleanly. Adding a `listCats` entry must be mirrored in `build.js` (both the `listCats` object **and** `EXPECTED_CATS`).
- Cells/items expose fields via child spans: `.he` (Hebrew w/ nikud), `.tr` (transliteration), `.fr` (French). Table extraction is **positional** ŌĆö it reads fixed column counts and indexes (Verbes needs Ōēź5 columns ŌåÆ il/elle/ils/elles forms; Adjectifs Ōēź4; Noms Ōēź3 where column 2 is the gender `m`/`f` and column 3 the plural).
- A `<li>` can carry `data-fr-court` (short French used on the card instead of the notebook's long `.fr`) and `data-note` (precision shown under the answer). This is how card-specific wording lives in the notebook rather than in app code.
- Grammar-only sections (racine, pass├®, futur, binyanim, article, smikhutŌĆ”) have no `.count` mapping into cards and are intentionally excluded from flashcards.

The card object schema produced (and embedded in the standalone file) is:
`{cat, he, tr, fr, note?, he_plain, forms?:[{he, tr, label, he_plain}], genre?}` where `he_plain` is `he` with nikud stripped (`stripNikud`). Table-derived cards have `tr:''`; the UI falls back to `he2tr(card.he)` for display.

Two guards catch silent extraction breakage: `init()` in `index.html` warns (console + setup screen) when an expected category yields 0 cards, and `node build.js` prints per-section counts and exits non-zero on an empty section. After editing the notebook: run `node build.js`, check the counts, and verify in the browser that the loader shows the expected "N mots charg├®s".

## Answer checking (the flashcards' hardest logic)

`checkAnswer` (near line 857) grades typed answers leniently:
- **French direction** (`he2fr`): `normFr` strips accents/case, and `frVariants` splits the French field on `/`, `,`, parentheses, and articles so multiple phrasings pass.
- **Hebrew direction** (`fr2he`): accepts either real Hebrew (typed via the built-in virtual Israeli-layout keyboard, rows defined near line 956) **or** loose French-style transliteration. Transliteration is normalized to a canonical key by `trKey` (folds `phŌåÆf`, `kh/chŌåÆh`, `qŌåÆk`, `wŌåÆv`, `tz/ts`, `ouŌåÆu`, strips apostrophes, collapses doubles) and compared against `he2tr(card.he)` ŌĆö a nikud-driven HebrewŌåÆtransliteration generator ŌĆö with a small Levenshtein (`editDist`) tolerance. When changing acceptance behavior, edit `trKey`/`he2tr` together since they must agree on the canonical form ŌĆö and note `he2tr` is also used for **display** wherever a card has no notebook `tr`.

**Transliteration standard** (all notebook `.tr` values and `he2tr` output follow it): `kh` = khaf without dagesh, `ch` = het (final furtive patach ŌåÆ `ach`), `ts` = tsadi, `'` = ayin everywhere (`'ivrit`, `be'er`, `yode'a`) and alef between two vowels (`tsme'ah`), final hey kept (`atah`, `zeh`), `ei` for tsere/segol+yud (`beit`), `u`/`f`/`k`/`v` (never `ou`/`ph`/`q`/`w`). Initial cluster shva is written only when heard (`gdolim` but `ledaber`) ŌĆö the notebook is authoritative there; don't blindly regenerate hand-written `.tr` values from `he2tr`. Thanks to `trKey`'s folding, the standard is display-only: typing stays tolerant to all variants.

## UI settings model

The setup screen uses `.chip` segmented toggles carrying `data-*` attributes (`data-mode`, `data-dir`, `data-script`, `data-order`, `data-audio`), wired by `segPick`. Modes (`state.mode`): `cards` (flip), `input` (typed), `quiz` (QCM ŌĆö 4 choices, distractors from `pickDistractors`, same category first). Each mode has its own zone in the study screen (`#controls-cards` / `#input-zone` / `#quiz-zone`) and a `setup*Card()` that toggles the `input-mode`/`quiz-mode` body classes; `render()` branches on `state.mode`. Direction heŌåÆfr / frŌåÆhe; script nikud / plain / cursive; audio uses the browser's Hebrew SpeechSynthesis voice.

**Spaced repetition (Leitner)**: `recordResult(card, good)` is called from *every* answer path (cards `answer`, input `submitAnswer`/`skipAnswer`, `quizPick`) and persists a per-card box/due record to `localStorage` under `srs_v1` (card identity = `cat|he_plain`). The setup screen's "R├®vision du jour" button (`startReview`) builds a session from `dueCards()`; `refreshSrsUi()` (called at the end of `buildChips()`, so it runs in both the online and standalone boot paths) updates the due count and the mastery bar. Intervals in `SRS_INTERVALS`. This layer is invisible to `build.js` ŌĆö it's pure app state.
