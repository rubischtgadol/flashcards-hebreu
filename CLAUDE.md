# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A French-language toolkit for learning modern Hebrew, deployed as static files on GitHub Pages at `https://rubischtgadol.github.io/flashcards-hebreu/`. There are **no dependencies, no tests, no package manager** — every deployed file is a single self-contained HTML document with inline CSS and vanilla JS. The only tooling is `build.js` (dev-only, zero-dependency Node script, not deployed). To develop, open the files in a browser; to deploy, replace files on the `main` branch (GitHub Pages redeploys automatically, same URLs).

Important: `index.html` uses `fetch()`, so it must be served over HTTP, not opened as a `file://`. Run a local server from the repo root (e.g. `python3 -m http.server`) and open `http://localhost:8000/`.

## The files

- **`vocabulaire_hebreu.html`** — the grammar + vocabulary notebook. This is the **single source of truth** for all content. Editing vocabulary means editing this file.
- **`index.html`** — the online flashcards app (the main page). It does **not** contain vocabulary; at load it fetches `vocabulaire_hebreu.html` and extracts the cards at runtime. It is also the **source of the standalone file**: the network-only code (fetch + runtime extraction) is fenced by `BUILD:ONLINE-ONLY` marker comments that `build.js` relies on.
- **`flashcards_hebreu.html`** — the standalone offline flashcards. **100% generated by `build.js` — never edit it by hand.** It is `index.html` with the `BUILD:ONLINE-ONLY` block replaced by an embedded `const CARDS = [...]` snapshot and a direct boot (no loader).
- **`build.js`** — regenerates `flashcards_hebreu.html` from the two files above, printing the per-section card count and failing if an expected section drops to 0. `node build.js --check` verifies without writing. **Run it after any edit to the notebook or to `index.html`.** It re-implements `extractCards()` with regex parsing (no DOM in Node) — any change to `extractCards()` in `index.html` must be mirrored there, and `--check` byte-compares so drift is caught.

## The extraction coupling (most important thing to understand)

`index.html`'s `extractCards(doc)` (near line 1024) scrapes the notebook's DOM. Its parsing is tightly coupled to the notebook's markup, so edits to `vocabulaire_hebreu.html` can silently drop cards if they break these assumptions:

- Sections are found by `<h2>` elements containing a `<span class="count">LABEL</span>`. The **exact `LABEL` text** is the key used throughout `extractCards` (e.g. `'Verbes'`, `'Adjectifs'`, `'Noms'`, `'Pronoms personnels'`, `'Nombres (0–10)'`). Renaming a count label detaches its section from extraction.
- Word entries live in either `<table><tbody><tr>` rows (Verbes, Adjectifs, Noms) or `<ul class="word-list"><li>` items (pronouns, prepositions, numbers, expressions, etc. — see the `listCats` map).
- Cells/items expose fields via child spans: `.he` (Hebrew w/ nikud), `.tr` (transliteration), `.fr` (French). Table extraction is **positional** — it reads fixed column counts and indexes (Verbes needs ≥5 columns → il/elle/ils/elles forms; Adjectifs ≥4; Noms ≥3 where column 2 is the gender `m`/`f` and column 3 the plural).
- A `<li>` can carry `data-fr-court` (short French used on the card instead of the notebook's long `.fr`) and `data-note` (precision shown under the answer). This is how card-specific wording lives in the notebook rather than in app code.
- Grammar-only sections (racine, passé, futur, binyanim, article, smikhut…) have no `.count` mapping into cards and are intentionally excluded from flashcards.

The card object schema produced (and embedded in the standalone file) is:
`{cat, he, tr, fr, note?, he_plain, forms?:[{he, tr, label, he_plain}], genre?}` where `he_plain` is `he` with nikud stripped (`stripNikud`). Table-derived cards have `tr:''`; the UI falls back to `he2tr(card.he)` for display.

Two guards catch silent extraction breakage: `init()` in `index.html` warns (console + setup screen) when an expected category yields 0 cards, and `node build.js` prints per-section counts and exits non-zero on an empty section. After editing the notebook: run `node build.js`, check the counts, and verify in the browser that the loader shows the expected "N mots chargés".

## Answer checking (the flashcards' hardest logic)

`checkAnswer` (near line 826) grades typed answers leniently:
- **French direction** (`he2fr`): `normFr` strips accents/case, and `frVariants` splits the French field on `/`, `,`, parentheses, and articles so multiple phrasings pass.
- **Hebrew direction** (`fr2he`): accepts either real Hebrew (typed via the built-in virtual Israeli-layout keyboard, rows defined near line 925) **or** loose French-style transliteration. Transliteration is normalized to a canonical key by `trKey` (folds `ph→f`, `kh/ch→h`, `q→k`, `w→v`, `tz/ts`, `ou→u`, collapses doubles) and compared against `he2tr(card.he)` — a nikud-driven Hebrew→transliteration generator (handles word-initial vocal shva → `e` and furtive patach on final `ח` → `ach`) — with a small Levenshtein (`editDist`) tolerance. When changing acceptance behavior, edit `trKey`/`he2tr` together since they must agree on the canonical form — and note `he2tr` is also used for **display** wherever a card has no notebook `tr`.

## UI settings model

The setup screen uses `.chip` segmented toggles carrying `data-*` attributes (`data-mode`, `data-dir`, `data-script`, `data-order`, `data-audio`), wired by `segPick`. Modes: cards vs. typed input; direction he→fr / fr→he; script nikud / plain / cursive; audio uses the browser's Hebrew SpeechSynthesis voice.

## Content note

All UI text and documentation is in **French**. Match that when editing user-facing strings.
