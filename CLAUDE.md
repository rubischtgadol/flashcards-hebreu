# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A French-language toolkit for learning modern Hebrew, deployed as static files on GitHub Pages at `https://rubischtgadol.github.io/flashcards-hebreu/`. There are **no dependencies, no tests, no package manager** ŌĆö every deployed file is a single self-contained HTML document with inline CSS and vanilla JS. The only tooling is `build.js` and `verifie_exemples.js` (dev-only, zero-dependency Node scripts, not deployed). To develop, open the files in a browser; to deploy, replace files on the `main` branch (GitHub Pages redeploys automatically, same URLs).

Important: `app.html` uses `fetch()`, so it must be served over HTTP, not opened as a `file://`. Run a local server from the repo root (e.g. `python3 -m http.server`) and open `http://localhost:8000/`.

Verification tooling in this WSL environment: visual checks run through **Playwright + real WebKit** (Safari's engine, matching the target iPhone ŌĆö `devices['iPhone 16 Pro']` for mobile), pure logic through jsdom; the system headless Chrome hangs in WSL2 ŌĆö never use it. Browsers persist in `~/.cache/ms-playwright`; only the npm `playwright` lib needs reinstalling per session (in the scratchpad). Operational details in TODO.md ┬¦ Outillage.

## The extraction coupling (most important thing to understand)

`app.html`'s `extractCards(doc)` (near line 2004) scrapes the notebook's DOM. Its parsing is tightly coupled to the notebook's markup, so edits to `vocabulaire_hebreu.html` can silently drop cards if they break these assumptions:

- Sections are found by `<h2>` elements containing a `<span class="count">LABEL</span>`. The **exact `LABEL` text** is the key used throughout `extractCards` (e.g. `'Verbes'`, `'Adjectifs'`, `'Noms'`, `'Pronoms personnels'`, `'Nombres (0ŌĆō10)'`). Renaming a count label detaches its section from extraction.
- Word entries live in either `<table><tbody><tr>` rows (Verbes, Adjectifs, Noms) or `<ul class="word-list"><li>` items (pronouns, prepositions, numbers, expressions, **Phrases**, etc. ŌĆö see the `listCats` map). The `Phrases` section holds full example sentences (cat `Phrases`); they flow through every mode as ordinary cards and get a smaller `.big-he.phrase`/`.big-fr.phrase` size so long sentences wrap cleanly. Adding a `listCats` entry must be mirrored in `build.js` (both the `listCats` object **and** `EXPECTED_CATS`).
- Cells/items expose fields via child spans: `.he` (Hebrew w/ nikud), `.tr` (transliteration), `.fr` (French). Table extraction is **positional** ŌĆö it reads fixed column counts and indexes (Verbes needs Ōēź5 columns ŌåÆ il/elle/ils/elles forms; Adjectifs Ōēź4; Noms Ōēź3 where column 2 is the gender `m`/`f` and column 3 the plural).
- A `<li>` can carry `data-fr-court` (short French used on the card instead of the notebook's long `.fr`) and `data-note` (precision shown under the answer). This is how card-specific wording lives in the notebook rather than in app code.
- Every `<li>` **and** every `<tr>` of the three tables can carry `data-niveau="A1"ŌĆ”"C2"` (fine-grained CEFR level, classification method in ARCHITECTURE.md ┬¦4). Optional by design: an unclassified word ignores the app's level filter and stays visible everywhere. `build.js` prints per-level counts and fails if a level in `EXPECTED_LEVELS` drops to 0.
- A word can carry usage examples: a nested `<ul class="exemples"><li>` (spans `.he`/`.tr`/`.fr`) inside the word's `<li>`, or at the **end** of the first `<td>` for table rows (order matters ŌĆö extraction reads the *first* `.he`/`.fr` of the fragment, so the word's own spans must come first). ŌÜĀ’ĖÅ Nested `<li>` break naive regex: `lisOf` in build.js does depth-aware scanning for top-level `<li>` only, and app.html uses the `ul.word-list > li` direct-child selector ŌĆö keep both properties when touching the parsers. Property insertion order in card objects must stay identical between the two extractors (`--check` byte-compares the embedded snapshot).
- Grammar-only sections (racine, pass├®, futur, binyanim, article, smikhutŌĆ”) have no `.count` mapping into cards and are intentionally excluded from flashcards.

The card object schema produced (and embedded in the standalone file) is:
`{cat, he, tr, fr, note?, niveau?, exemples?:[{he, tr, fr, he_plain}], he_plain, forms?:[{he, tr, label, he_plain}], genre?}` where `he_plain` is `he` with nikud stripped (`stripNikud`) and `niveau` is the raw CEFR value from `data-niveau` (absent if unclassified). Table-derived cards have `tr:''`; the UI falls back to `he2tr(card.he)` for display.

Two guards catch silent extraction breakage: `init()` in `app.html` warns (console + setup screen) when an expected category yields 0 cards, and `node build.js` prints per-section counts and exits non-zero on an empty section. After editing the notebook: run `node build.js`, check the counts, and verify in the browser that the loader shows the expected "N mots charg├®s".

## Answer checking (the flashcards' hardest logic)

`checkAnswer` (near line 1679) grades typed answers leniently and returns `'exact'`, `'almost'`, or `false` ŌĆö truthy means accepted, and `'almost'` (accepted only via the `editDist` tolerance) makes `showInputFeedback` display the ┬½ Ō£ō Presque ! La forme exacte : ┬╗ verdict (green, attempt shown un-struck). Feedback kinds are `'ok' | 'almost' | 'no' | 'skip'`; `fixVerdict` treats `ok`/`almost` as "was counted good". Details:

- **French direction** (`he2fr`): `normFr` strips accents/case, and `frVariants` splits the French field on `/`, `,`, parentheses, and articles so multiple phrasings pass.
- **Hebrew direction** (`fr2he`): accepts either real Hebrew (typed via the built-in virtual Israeli-layout keyboard, rows defined near line 1824 ŌĆö folded behind the ┬½ Clavier h├®breu ┬╗ toggle, and **hidden entirely on touch devices** via `@media (pointer:coarse)` with `!important`, a 2026-07-19 decision: iOS provides a native Hebrew keyboard, the virtual one only serves physical AZERTY desktop keyboards) **or** loose French-style transliteration. Transliteration is normalized to a canonical key by `trKey` (folds `phŌåÆf`, `kh/chŌåÆh`, `qŌåÆk`, `wŌåÆv`, `tz/ts`, `ouŌåÆu`, strips apostrophes, collapses doubles) and compared against `he2tr(card.he)` ŌĆö a nikud-driven HebrewŌåÆtransliteration generator ŌĆö with a small Levenshtein (`editDist`) tolerance. When changing acceptance behavior, edit `trKey`/`he2tr` together since they must agree on the canonical form ŌĆö and note `he2tr` is also used for **display** wherever a card has no notebook `tr`.

**Transliteration standard** (all notebook `.tr` values and `he2tr` output follow it): `kh` = khaf without dagesh, `ch` = het (final furtive patach ŌåÆ `ach`), `ts` = tsadi, `'` = ayin everywhere (`'ivrit`, `be'er`, `yode'a`) and alef between two vowels (`tsme'ah`), final hey kept (`atah`, `zeh`), `ei` for tsere/segol+yud (`beit`), `u`/`f`/`k`/`v` (never `ou`/`ph`/`q`/`w`). Initial cluster shva is written only when heard (`gdolim` but `ledaber`) ŌĆö the notebook is authoritative there; don't blindly regenerate hand-written `.tr` values from `he2tr`. Thanks to `trKey`'s folding, the standard is display-only: typing stays tolerant to all variants.

## UI settings model

The setup screen opens on the ┬½ R├®vision du jour ┬╗ card (the daily action leads); the search drawer sits right below the mastery bar, before the categories. It uses `.chip` segmented toggles carrying `data-*` attributes (`data-mode`, `data-dir`, `data-script`, `data-order`, `data-audio`, `data-len`), wired in a loop over `SEG_KEYS` by `segPick(container, key, btn)`. Every `.seg` (and `#cats`) is a `role="group"` labelled by its `<h2>` (`aria-labelledby`, notes via `aria-describedby`). The set-once groups (Ordre/Longueur/Prononciation) live inside the native `<details class="adv" id="adv">` fold, closed by default; under `pointer:coarse` the `#start` button is `position:sticky` at the bottom **only while enabled** (`.start:enabled` ŌĆö a 2026-07-19 critique fix: sticky *and* disabled, it floated a translucent gold slab over four category chips and ate their taps, `elementFromPoint` returning `start`). `#start:disabled` therefore carries a full opaque skin (`background:none`, `--line` border, `--ink-dim` text) instead of the old `opacity:.4` on the gold gradient ŌĆö gold must never appear on an inactive state (DESIGN.md, r├©gle de la lampe), and an opacity let page content show through the button. The empty-selection hint `#start-hint` sits **above** it in the DOM so it stays visible while stuck, and carries `role="status"` so its three messages are announced. Modes (`state.mode`): `cards` (flip), `input` (typed), `quiz` (QCM ŌĆö 4 choices, distractors from `pickDistractors`, same category first, **rejecting any candidate whose French variants are within Levenshtein 1 of already-kept options** so choices are never quasi-synonyms; a relaxed last-resort pass still guarantees 4 options). Each mode has its own zone in the study screen (`#controls-cards` / `#input-zone` / `#quiz-zone`) and a `setup*Card()` that toggles the `input-mode`/`quiz-mode` body classes; `render()` branches on `state.mode`. Direction heŌåÆfr / frŌåÆhe; script nikud / plain / cursive. Audio uses the browser's Hebrew SpeechSynthesis voice: `'click'` means **only** the speaker button triggers speech (all answer paths honor this; the **P key** counts as a deliberate trigger and follows the speaker button's visibility rules), `'auto'` speaks on render/reveal; with no Hebrew voice, `reflectVoiceUi()` sets `body.no-he-voice` (speaker buttons + `.spk-hint` hidden) **and** disables the `#audio` chips; when a voice is found, `#audio-note` displays **its real name** (┬½ Voix h├®bra├»que d├®tect├®e Ō£ō ŌĆö ŌĆ” ┬╗ ŌĆö the diagnostic for the robotic-voice issue). Keyboard parity: cards Space/ŌåÉ/ŌåÆ, input Enter, quiz **1ŌĆō4 to pick + Enter/Space for next** (a focused button keeps native handling), and **C** presses the current mode's correction button (┬½ Corriger ┬╗ / ┬½ Annuler la derni├©re r├®ponse ┬╗) when it is visible and enabled ŌĆö both P and C ignore Ctrl/Cmd/Alt combos; `#kbd-hint` content adapts per mode.
